## 🔴 P0 — блокеры при росте трафика

### [ ] Server-side пагинация на `GET /admin/users`

**Когда:** при ~1000+ зарегистрированных пользователях.

**Почему важно:** сейчас бэкенд (`src/api/routes/admin/users.py`) и
сервис (`AdminUserService.get_users`) тянут **всех** пользователей
из Keycloak за один запрос, фильтр и поиск делается в памяти Python.
Для админки фронт делает то же — `/users`-страница пагинирует
**уже загруженный** массив (`users.slice(start, end)` в
`UserList.tsx`).

**Цена бездействия:** при 5000+ юзеров запрос Keycloak Admin REST
начнёт занимать секунды, а потом отваливаться по таймауту.

**План фикса:**
1. Бэкенд: добавить `limit: int = 50, offset: int = 0` query-параметры
   в `GET /admin/users` и пробросить их в Keycloak Admin REST
   (`first` / `max` параметры).
2. Возвращать `{items: list[UserDTO], total: int, limit: int, offset: int}`
   вместо плоского списка. Считать `total` через отдельный
   `get_users_count()` (Keycloak Admin REST → `/users/count`).
3. Фронт: переписать `userStore` и `UserList` на server-side
   pagination. Дёргать `/admin/users?limit=50&offset=0` при смене
   страницы. Поиск/фильтры тоже серверные.

**Расчёт:** до 5000 юзеров (порог поставлен 09.05.2026) — текущая
архитектура работоспособна с in-memory обработкой. После — обязательная
переработка.

## 🟡 P1 — функциональные расширения

### [ ] Передача `grade` из мобильного приложения при регистрации

**Контекст:** на бэкенде поле `grade` (5..11) добавлено в
`UserDTO` + `KeycloakAttributesDTO` коммитом от 09.05.2026.
Само значение в Keycloak пока **не пишется**, потому что:
- `RegistrationCompleteDTO` (`POST /auth/registration/complete`) не
  принимает `grade`.
- Мобильное приложение не отправляет этот параметр.

Старые юзеры при этом останутся с `grade = null` навсегда —
бэкфил отдельной задачей (либо ручной через `PATCH /admin/users/{id}`,
либо одноразовый скрипт по `phone → ask user to re-confirm grade`).

**План:**
1. Бэкенд: добавить `grade: int | None = Field(None, ge=1, le=11)` в
   `RegistrationCompleteDTO` + сохранить в `KeycloakAttributes` при
   создании юзера.
2. Mobile (Flutter): на онбординг-экране «В каком классе ты учишься?»
   уже спрашивается значение — пробросить его до
   `register_user_remote_source.dart#completeRegistration`.

### [ ] Редактирование `grade` и `roles` из админки

**Контекст:** `AdminUserUpdateDTO` сейчас умеет менять `name`, `email`,
`phone`, `password`, `allowed_subject_ids`, `is_active`. **НЕ умеет**
менять `grade` и `roles`.

**Когда понадобится:** когда админу нужно «переназначить родителя в
учителя», или поправить класс ученика после ошибки регистрации.

**План:** расширить `AdminUserUpdateDTO` + сервисный слой + добавить
поля в формы редактирования (UserForm) на фронте. Аккуратно с `roles` —
менять keycloak-роль это операция в Identity Provider, не просто
attribute write.

### [ ] Bulk-действия (масс-блокировка / удаление)

**Когда понадобится:** массовый бан спамеров, очистка тестовых
аккаунтов после QA-сессии.

**План:** `selectable=true` уже стоит в `ListTable`, нужно только
обработчик `onBulkAction(ids, action)` на фронте + бэкенд-эндпоинт
`POST /admin/users/bulk` с body `{ids: [...], action: "block" | "delete"}`.

## 🟢 P2 — UX-улучшения

### [ ] Карточка пользователя — больше данных

Сейчас `UserDetail` показывает только то что есть в `UserDTO`. Хорошо
бы добавить:

- **История тестов** — сколько ЕНТ/тренировок прошёл, средний балл
  (есть в `EntAttempt` таблице, нужен `GET /admin/users/{id}/attempts`).
- **Активность** — последние 30 дней по дням (heat-map), время в
  приложении (есть `TimeInAppEntity`).
- **Платежи** — история транзакций FreedomPay (если у юзера PRO).
- **Push-токены** — список устройств юзера (FCM tokens) с возможностью
  отозвать.

### [ ] Колонки таблицы — настраиваемые

Сейчас колонки захардкожены в `UserList`. Чтобы каждый админ мог
показать/скрыть поля по своему вкусу — вынести в localStorage или в
профиль админа.

### [ ] Экспорт списка в CSV

Маркетинг будет просить выгрузку списка всех пользователей с фильтром
(например, «все ученики 10 класса с подпиской PRO»). Кнопка `Download`
с таким же набором фильтров что и таблица. Делать на бэкенде, чтобы не
тянуть весь список в браузер.

### [ ] Индикация админа онлайн / последний логин

Колонка «Последний вход» (Keycloak хранит в session-table). Поможет
отличить активных юзеров от мёртвых регистраций.

### [ ] Семейные связки в карточке родителя

Когда фича «Семья» доедет до прода — в карточке родителя показывать
список его детей с быстрыми ссылками на их карточки (и наоборот, в
карточке ребёнка — родителя).

## 📊 Маркетинг-раздел — будущие улучшения

Раздел `/marketing` запущен 09.05.2026 с базовым набором метрик
(KPI-cards, регистрации по месяцам, retention, доходы, аудитория,
эффективность контента). Что докрутить дальше:

### [ ] UTM-источники регистраций

**Что:** прокинуть `utm_source / utm_medium / utm_campaign` через
landing → app store → mobile app → бэкенд → сохранить в Keycloak
attributes юзера → агрегировать в дэшборде.

**Цена бездействия:** сейчас невозможно понять «откуда пришли все
эти 5к юзеров» — ни Google ads, ни Instagram, ни сарафан не различимы.
Это самая большая дыра в текущей атрибуции.

### [ ] Воронка регистрации

**Что:** events `/auth/code/request` → `/auth/code/check` →
`/auth/registration/complete`. У бэкенда они уже логируются (видно в
Railway logs). Превратить в воронку: «Запросили SMS → Ввели код →
Завершили регистрацию» с конверсиями на каждом шаге.

**План:** добавить агрегатный endpoint `/admin/analytics/funnel/registration`
который читает события из логов или новой таблицы `auth_events`.

### [ ] Когортный анализ retention

**Что:** разрезать retention по когортам (юзеры зарегавшиеся в
конкретный месяц) и смотреть как они себя ведут на D1/W1/M1/M3.
Сейчас retention — общая цифра без когортной разбивки во времени.

### [ ] LTV / ARPU

**Что:** рассчитать средний доход на пользователя за всё время и
поделить на роль/класс/источник трафика.

**Цена бездействия:** не знаем сколько стоит привлекать платного
юзера vs какие маркетинг-каналы окупаются.

### [ ] Date-range фильтр глобальный для всего раздела

**Что:** dropdown `Сегодня / 7 дней / 30 дней / 90 дней / Всё время`
в шапке Marketing dashboard. Все секции реагируют. Сейчас все
endpoints возвращают «всё время».

### [ ] Push-engagement

**Что:** какой % юзеров получивших push открыл его. FCM делает
аналитику на стороне Firebase, нужно подтянуть в дэшборд через
Firebase Admin SDK.

### [ ] Promocode-эффективность

**Что:** какие промокоды конвертили больше всего, какой средний LTV у
юзера привязанного к промокоду vs органика. Нужен JOIN на
`promocode_usages` в бэкенде.

### [ ] Trial → Paid конверсия

**Что:** % юзеров кто использовал бесплатный trial и потом купил
платную подписку. У бэкенда есть `used_trial` + `plan` — собрать
конверсию через `/admin/analytics/trial_conversion`.
